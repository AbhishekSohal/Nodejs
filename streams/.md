3. What is Buffering?
- Buffering is temporarily storing data in a physical memory space (a "buffer") before it is processed or sent.
- Without streams (e.g., `fs.readFile`): Node loads the entire file into memory at once. If the file is 1GB, it takes 1GB of RAM.
- With streams: Node uses small internal buffers (default is 64KB for read streams) to process the data chunk-by-chunk. Once a chunk is sent, the buffer is cleared and reused, keeping memory usage constant and extremely low.

4. Flow Modes (Paused vs. Flowing)
- **Paused Mode (Default):** The stream does not push data automatically. Data sits in the internal buffer until you explicitly call `stream.read()`.
- **Flowing Mode:** Data is read automatically and pushed as fast as possible. Triggered by adding a `'data'` event listener, calling `stream.resume()`, or using `stream.pipe()`.
- You can switch between modes using `stream.pause()` and `stream.resume()`.

5. Backpressure & `.pipe()`
- **Backpressure:** Occurs when the readable stream reads data faster than the writable stream can write it. This would cause memory to bloat if unmanaged.
- **Auto-management by `.pipe()`:**
  1. If the writable stream's buffer is full, `write()` returns `false`.
  2. `.pipe()` automatically pauses the readable stream (`readable.pause()`).
  3. When the writable stream finishes writing and clears its buffer, it emits a `'drain'` event.
  4. `.pipe()` listens for `'drain'` and automatically resumes the readable stream (`readable.resume()`).

6. Key Stream Events
- **`'data'`**: Emitted by a readable stream when a chunk of data is ready to be consumed (switches stream to flowing mode).
- **`'end'`**: Emitted by a readable stream when there is no more data to read.
- **`'drain'`**: Emitted by a writable stream when its buffer is empty and it's ready for more data.
- **`'finish'`**: Emitted by a writable stream after `end()` is called and all buffered data has been written to the system.
- **`'error'`**: Emitted when something goes wrong. *Always listen to this event*, otherwise Node.js throws an unhandled exception and crashes.
- **`'close'`**: Emitted when the stream and its underlying resource (like a file) have been closed.

7. 1. Flow Modes (Paused vs. Flowing)
In Node.js, every Readable stream operates in one of two modes:

Mode	How it behaves	How to trigger it
Paused Mode (Default)	The stream does not read data automatically. Data sits in the internal buffer, and you must explicitly pull it out by calling stream.read().	Streams start in this mode. You can force it with stream.pause().
Flowing Mode	Data is read from the system automatically and pushed to your app as fast as possible using events or piping.	Add a 'data' event listener, call stream.resume(), or call stream.pipe().
2. How .pipe() Manages Backpressure Automatically
What is Backpressure? Backpressure occurs when the Readable stream reads data faster than the Writable stream (like a slow network client or a slow hard drive) can write it. If you don't control it, the data will pile up in your server's RAM, causing memory bloat.

How .pipe() saves you: When you do readable.pipe(writable), Node.js manages backpressure automatically behind the scenes:

readable starts sending chunks of data.
If writable cannot keep up and its internal buffer fills up, its .write() method returns false.
.pipe() catches this false value and automatically calls readable.pause(), stopping the flow of incoming data.
Once the writable stream has finished writing and emptied its buffer, it emits a 'drain' event.
.pipe() catches the 'drain' event and automatically calls readable.resume() to start the flow again.
Without .pipe(), you would have to write this pause/resume handshake logic manually!

3. Key Stream Events
Here is a list of the vital events emitted by streams:

Readable Stream Events:
'data': Emitted whenever the stream passes a chunk of data to the consumer. (Puts the stream into Flowing mode).
'end': Emitted when there is no more data to be read from the stream. (The source is completely consumed).
Writable Stream Events:
'drain': Emitted when the writable stream's buffer has cleared and it is ready to receive more data. (Used to resume a paused readable stream).
'finish': Emitted after you call writable.end() and all remaining buffered data has been successfully written to the underlying system (e.g. written to disk or sent over the network).
Shared Events (Readable & Writable):
'error': Emitted when something goes wrong during reading or writing (e.g., file not found, permission denied).
IMPORTANT

Always register an listener for 'error'. If a stream emits 'error' and there is no listener, the Node.js process will crash.

'close': Emitted when the stream and its underlying resource (like a file descriptor) have been closed. This is the absolute end of the stream lifecycle.
